# API Reference

Base path: `/api/v1`

Two namespaces live under it:

- **Core endpoints** — implemented as real Next.js App Router route handlers under `src/app/api/v1/`.
- **Module endpoints** — declared by each installed module's `module.json` `api[]` array, served by the `/api/v1/[...path]` catch-all that reads `ModuleApiRegistry` (generated by `scripts/generate-registry.ts`).

NextAuth's own handlers live at `/api/auth/...` (outside `/api/v1`). The public health probe is at `/api/health`. The OpenAPI spec is `/api/v1/openapi`.

---

## Authentication

Three mechanisms are accepted:

| Mechanism | Header / cookie | Issued by |
|-----------|-----------------|-----------|
| Session cookie | Auth.js JWT cookie set by NextAuth | Logging in through the web UI |
| API key | `x-api-key: <key>` | **Admin > API Keys** (scoped per permission, see [docs/ADMIN_GUIDE.md](ADMIN_GUIDE.md#api-keys)) |
| Internal | `x-internal-request: $CSRF_INTERNAL_SECRET` | Server-to-server callers; bypasses the CSRF origin guard |

Most core endpoints accept session cookies only. Routes that admit API keys (e.g. `POST /admin/cron`) call `validateApiKey(rawKey, requiredPermission)`.

Inside a handler, use the helpers from core:

```typescript
import { auth } from "@/core/lib/auth";
import { isAdmin, isStaff, hasPermission } from "@/core/lib/permissions";

const session = await auth();
if (!session?.user?.id) return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
if (!(await isAdmin(session.user.id))) return NextResponse.json({ error: "Forbidden" }, { status: 403 });
```

## Response Envelope

The canonical envelope is defined in `src/core/lib/api-utils.ts`:

```typescript
import { apiSuccess, apiError, apiPaginated } from "@/core/lib/api-utils";

return apiSuccess({ id, name });                       // 200 { ok: true, data: { ... } }
return apiError("Validation failed", 400, { code: "invalid_input", details: parsed.error.issues });
return apiPaginated(items, total, page, limit);        // 200 { ok: true, data: [...], pagination: { ... } }
```

Successful responses:

```json
{ "ok": true, "data": { /* T */ }, "pagination": { "page": 1, "limit": 20, "total": 47, "pages": 3, "hasMore": true } }
```

Failures:

```json
{ "ok": false, "error": "human message", "code": "machine_code", "details": null }
```

Validation errors (Zod 4 — read `.issues`, not `.errors`):

```json
{ "ok": false, "error": "Validation failed", "code": "invalid_input", "details": [ /* Zod issues */ ] }
```

`devOnlyDetail(err)` strips internal error messages in production so DB hostnames, file paths, and stack frames never leak.

Legacy core handlers still use ad-hoc `{ error }` / `{ data }` shapes — new code MUST use the helpers above. Modules are free to do the same; consistency is recommended.

Every response includes an `x-correlation-id` header (set by the request logger) for log correlation.

## Security Baseline

| Layer | Detail |
|-------|--------|
| Rate limiting | Redis-backed sliding window (memory fallback). Per-role multipliers via the `rate_limit_role_multipliers` Setting key. Trusted-proxy aware via `getClientIP(headers)` + `TRUSTED_PROXY_IPS`. |
| CSRF | Origin check enforced by the proxy middleware on mutating routes. Server-to-server callers send `x-internal-request: $CSRF_INTERNAL_SECRET`. Allowed origins also come from `AUTH_URL`, `NEXTAUTH_URL`, `NEXT_PUBLIC_APP_URL`, plus extras from `CSRF_ALLOWED_ORIGINS`. |
| Session | Auth.js v5 JWT. 24 h `maxAge`, 1 h `updateAge` (role/ban/permission changes propagate to dormant sessions within an hour). `__Secure-` / `__Host-` cookie prefixes and the `Secure` flag activate when `AUTH_URL` starts with `https://`. |
| Passwords | bcrypt. Policy enforced by `password-policy.ts`. HaveIBeenPwned k-anon breach check. |
| Account lockout | Progressive per-account lockout (`account-lockout.ts`). 2FA failures count toward lockout. |
| File upload | Content-type sniffed, extension whitelist, size limit. Images piped through sharp; SVGs sanitized. |
| Activity log | `logActivity({ userId, action, entity, entityId, metadata })` writes to `ActivityLog`. Impersonation sessions rewrite the actor to the real admin. |
| Replay protection | Inbound webhooks reject deliveries older than `WEBHOOK_REPLAY_WINDOW_MS` (default 5 minutes). HMAC comparison is constant-time. |

## Rate Limiting

The platform exposes three baseline profiles in `src/core/lib/rate-limit.ts`:

| Profile | Default | Window |
|---------|---------|--------|
| `auth` | 10 req | 15 min |
| `api` | 100 req | 1 min |
| `upload` | 20 req | 1 min |

Apply inside a handler:

```typescript
import { rateLimit, rateLimitForRole, getClientIP } from "@/core/lib/rate-limit";

const ip = getClientIP(request.headers);
const limit = await rateLimit("auth", ip);
if (!limit.allowed) return NextResponse.json({ error: "Rate limited" }, { status: 429, headers: limit.headers });
```

`rateLimitForRole(profile, userId, ip)` looks up the user's role and applies the multiplier from `rate_limit_role_multipliers`. Multipliers are read at runtime — no rebuild required when an admin tunes them.

When Redis is configured (`REDIS_URL`), counters are shared across PM2 workers. Without Redis, the rate limiter falls back to in-memory storage; in multi-worker deployments this is unsafe and `ALLOW_MEMORY_RATE_LIMIT=1` must be set explicitly or rate-limited routes return 503.

## OpenAPI Specification

```
GET /api/v1/openapi
```

Returns the static OpenAPI 3.0 spec regenerated by `scripts/generate-openapi.ts` (runs during `prebuild`). Modules contribute documentation through the `description` field in each `api[]` entry of their `module.json`.

Access policy:

- Admin session → full spec.
- Anyone else → 401 by default. Set `OPENAPI_PUBLIC=1` to expose without auth (off by default — the spec enumerates every admin route).

Interactive Swagger UI lives at `/api/docs`.

---

## Health

```
GET /api/health
```

Unauthenticated. Rate-limited to ~30 req/min/IP. Returns a structured snapshot of database, Redis, email queue, and scheduler health. HTTP status reflects severity: `200 ok`, `200 degraded`, `503 down`. Raw error messages are masked unless `HEALTH_DEBUG=1`.

## Module APIs

Module endpoints are mounted at `/api/v1/[...path]` via a catch-all that resolves `path[]` against `ModuleApiRegistry`. The registry is built at registry-generation time from every installed module's `api[]` array.

Declared in `module.json`:

```json
"api": [
    { "path": "/blog/articles",        "handler": "api/articles/route.ts",      "method": "ALL", "description": "List and create articles" },
    { "path": "/blog/articles/[id]",   "handler": "api/articles/[id]/route.ts", "description": "Get, update, or delete an article" },
    { "path": "/blog/stats",           "handler": "api/stats/route.ts" }
]
```

Behavior:

- Dynamic segments like `[id]` or `[...params]` are supported and surfaced to the handler via `ctx.params`.
- The `method` field restricts which HTTP verbs the catch-all forwards. Default `ALL`.
- The handler file is a normal Next.js route handler module — export `GET`, `POST`, `PUT`, `DELETE`, `PATCH`.
- A request to an endpoint whose module is disabled or uninstalled returns 404.
- The dispatcher returns 405 when the requested method isn't exported, 500 when the handler is missing despite the registry pointing at it.

A handler skeleton:

```typescript
import { NextRequest, NextResponse } from "next/server";
import { auth } from "@/core/lib/auth";
import { prisma } from "@/core/lib/db";
import { hasPermission } from "@/core/lib/permissions";
import { logActivity } from "@/core/lib/activity-log";

export async function GET() {
    const articles = await prisma.blogArticle.findMany({ where: { status: "published" }, take: 20 });
    return NextResponse.json({ articles });
}

export async function POST(request: NextRequest) {
    const session = await auth();
    if (!session?.user?.id) return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
    if (!(await hasPermission(session.user.id, "blog.manage"))) {
        return NextResponse.json({ error: "Forbidden" }, { status: 403 });
    }
    const body = await request.json();
    const article = await prisma.blogArticle.create({
        data: { title: body.title, content: body.content, authorId: session.user.id, status: "draft" },
    });
    await logActivity({ userId: session.user.id, action: "blog.article.create", entity: "blog.article", entityId: article.id });
    return NextResponse.json({ article }, { status: 201 });
}
```

Permission gating, validation, and rate limiting are the module's responsibility. The dispatcher only routes and method-checks.

## Webhook Receivers

Inbound webhooks are dispatched by `POST /api/v1/webhook/[provider]`. The dispatcher resolves `provider` against the modules' `webhookReceivers[]` declarations.

Declared in `module.json`:

```json
"webhookReceivers": [
    {
        "provider": "stripe",
        "handler": "hooks/webhook.ts",
        "signatureHeader": "stripe-signature",
        "secretEnv": "STRIPE_WEBHOOK_SECRET"
    }
]
```

Pre-handler behavior:

- The dispatcher requires the receiving module to be enabled — otherwise 404.
- When `signatureHeader` and `secretEnv` are set, HMAC-SHA256 verification runs against the raw body using `crypto.timingSafeEqual`. Failures return 401.
- Replay protection: deliveries older than `WEBHOOK_REPLAY_WINDOW_MS` (default 5 minutes) are rejected. Timestamp formats accepted: Unix seconds, Unix ms, ISO-8601.

Handler contract:

```typescript
export default async function handler(request: Request): Promise<{ status: number; body?: unknown }> {
    const event = await request.json();
    // … process the event …
    return { status: 200, body: { received: true } };
}
```

Provider-specific signature schemes (e.g. the Stripe `t=...,v1=...` format) can still be verified inside the handler — set `signatureHeader` to a fake value or omit it entirely.

## Cron Jobs

Modules declare cron handlers in `module.json`:

```json
"cronJobs": [
    { "id": "currency-rate-refresh", "schedule": "every-hour", "handler": "cron/refresh.ts" }
]
```

Valid schedule keywords: `every-minute`, `every-5-minutes`, `every-15-minutes`, `every-hour`, `every-day`, `every-week`, `every-month`.

Handlers default-export an async function returning `Promise<void>`. The scheduler tracks runs in the `CronRun` table (`lastRunAt`, `lastStatus`, `lastError`, `lastRunMs`, `nextRunAt`).

Operators trigger ticks two ways:

1. **In-process scheduler** — boots automatically when the server starts (see `bootstrapScheduler` in `src/core/lib/scheduler.ts`).
2. **External cron + API key** — issue an API key with the `cron:run` permission, then:

   ```
   POST /api/v1/admin/cron
   x-api-key: <key>
   ```

   The endpoint runs the core scheduled tasks once and reports `{ results, ran }`.

A single job can also be re-run on demand by an admin: `POST /api/v1/admin/cron/[key]/run`.

## Hooks (Action / Filter)

Modules can subscribe to events fired by core or other modules via `hookListeners` in `module.json`:

```json
"hookListeners": [
    { "hook": "user.registered", "type": "action", "handler": "hooks/on-user-registered.ts", "priority": 10 },
    { "hook": "post.content",    "type": "filter", "handler": "hooks/filter-content.ts",      "priority": 20 }
]
```

- `type: "action"` — fire-and-forget. Handler signature: `(payload, context?) => void | Promise<void>`.
- `type: "filter"` — value transformation. Handler signature: `(value, context?) => value | Promise<value>`.
- `priority` — lower runs earlier. Default 10. Async listeners are wrapped with a per-listener timeout (`HOOK_LISTENER_TIMEOUT_MS`, default 5 s).

Fire your own hooks from anywhere in a module:

```typescript
import { doAction, doActionAsync, applyFilters, applyFiltersAsync } from "@/core/lib/hooks";

doAction("blog.article.created", { articleId, authorId });
const html = applyFilters("post.content", rawHtml, { postId });
```

## Locale and Routing

Public APIs are not locale-scoped. The `/api/v1/...` prefix is locale-free; only page routes live under `/[locale]/`. Active locales are listed in `src/core/lib/i18n/config.ts` (`en`, `tr`). Translations are read at request time from the `Translation` DB table — there are no JSON locale files served from disk in production.
