# API Reference

Base URL: `/api/v1`

---

## Authentication

Requests are authenticated in two ways:

- **Session cookie** — set automatically when a user logs in via the web UI (Auth.js v5 / NextAuth).
- **API key** — pass in the `x-api-key` header. Create keys at Admin > API Keys with scoped permissions.

---

## OpenAPI Specification

```
GET /api/v1/openapi
```

Returns the generated OpenAPI spec for all installed module APIs. Also served as interactive docs at `/api/docs`. The spec is regenerated by `npx tsx scripts/generate-openapi.ts`. Module endpoints are documented via the `description` field in each `api[]` entry of their `module.json`.

## Security Baseline

| Layer | Detail |
|-------|--------|
| Rate limiting | Redis-backed shared counter with memory fallback. Per-role multipliers via the `rate_limit_role_multipliers` Setting key. Trusted proxy support via `getClientIP(headers)` + `TRUSTED_PROXY_IPS` env. |
| CSRF | Double-submit cookie + custom header (`csrf.ts`). All mutating core routes verify this. |
| Session | Auth.js v5 JWT. 24 h `maxAge`, 1 h `updateAge`. Cookies hardened (`__Secure-` / `__Host-` prefix + `Secure` flag) when `AUTH_URL` begins with `https://`. |
| Passwords | bcrypt. Policy enforced by `password-policy.ts`. Breach check via HaveIBeenPwned k-anon API. |
| Account lockout | Progressive per-account lockout (`account-lockout.ts`). 2FA failures count toward lockout. |
| File upload | Content-type sniffed, extension whitelist, size limit. Images piped through sharp; SVG sanitized. |
| Activity log | `logActivity()` in `activity-log.ts` — structured insert for every sensitive action. Impersonation sessions rewrite actor to the real admin. |

|--------|------|------|-------------|
| POST | `/auth/register` | None | Register a new user account |

| POST | `/auth/reset-password` | None | Consume a reset token and set a new password |

## Module APIs

Module endpoints are mounted at `/api/v1/[...path]` via a catch-all that reads `ModuleApiRegistry` (generated from each module's `api[]` entries). They exist only when the module is installed and enabled.

Examples from installed modules:

```
GET  /api/v1/blog/articles
POST /api/v1/blog/articles
GET  /api/v1/blog/articles/[id]
GET  /api/v1/store/products
POST /api/v1/store/checkout
GET  /api/v1/store/orders
```

Permission gating is the module's responsibility — use `hasPermission` / `isAdmin` inside the handler. Module APIs are documented in full in the generated OpenAPI spec at `/api/v1/openapi`.

## Inbound Webhooks

```
POST /api/v1/webhook/[provider]
```

Routes to the module that declared `webhookReceivers[].provider` matching the URL segment. When `signatureHeader` and `secretEnv` are configured on the receiver, HMAC verification runs before the handler.

Example: `POST /api/v1/webhook/stripe` routes to the `stripe-gateway` module's webhook handler, which verifies the `stripe-signature` header against `STRIPE_WEBHOOK_SECRET`.
